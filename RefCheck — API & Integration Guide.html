API Overview

Integrating RefCheck into your workflow

RefCheck provides a RESTful JSON API that allows developers to integrate its reference-checking capabilities with other systems (like Applicant Tracking Systems or custom HR portals) and to automate key steps. The API supports creating reference checks, listing their statuses, retrieving completed reports, and receiving callbacks when checks are done. By using these endpoints and webhooks, an organization can seamlessly plug RefCheck into their existing hiring workflow:

  • **Automate reference requests:** For example, when a candidate in your ATS moves to a ‚ÄúReference Check‚Äù stage, you can have a trigger to call RefCheck‚Äôs API to create a new reference check for them, sending out emails instantly.
  • **Retrieve results programmatically:** Instead of manually logging into RefCheck to view outcomes, your system can pull the summarized report or raw feedback via the API and display it in your internal dashboard or attach it to the candidate‚Äôs ATS profile.
  • **Receive push notifications:** Using webhooks, RefCheck can notify your application the moment all references have submitted feedback (or if a reference declines or a deadline passes), so you can take immediate action (for instance, automatically move the candidate‚Äôs stage forward or alert a recruiter).

The API is organized around familiar REST principles. It uses predictable URLs and standard HTTP response codes to indicate errors or success. Request and response bodies are in JSON format. If you‚Äôve used modern web APIs, integrating with RefCheck should feel straightforward. In this guide, we‚Äôll cover how to authenticate with the API, describe the key endpoints and their usage with examples, and explain how to set up webhooks for real-time integration with external systems like ATS or messaging tools.

**Base URL:** All API requests are made to a base endpoint, for example https://api.refcheck.com/v1/ (for production). The examples below assume this base. We version our API (v1 in this case) to allow future enhancements without breaking existing integrations. Wherever possible, new features will be added in a backward-compatible way, and any breaking changes would result in a new version.

Authentication & Security

Secure access to the RefCheck API

The RefCheck API uses token-based authentication over HTTPS. Only authorized users or systems should be able to create reference checks or read their results. There are two ways to authenticate, depending on your integration scenario:

  • User Session Token (OAuth/JWT): If you‚Äôre calling the API from a logged-in context (e.g., your own front-end after the recruiter has logged in, or from your server on behalf of a user), you will include an Authorization header with a bearer token. For instance, after login, the recruiter‚Äôs session may provide a JWT token. Use it as:
    Authorization: Bearer {your_token}
    The token inherently carries the user‚Äôs identity and permissions. The API will determine which company‚Äôs data the user can access based on it. This approach is typically used if RefCheck is integrated within your web app and uses RefCheck‚Äôs built-in authentication or SSO.
  • API Key (Service-to-Service): For server-to-server integrations or background services (like an ATS webhook calling RefCheck), you can use a static API key. You can obtain an API key from the RefCheck admin settings (or it can be provided during onboarding). This key is specific to your company account and has predefined privileges. Include it similarly:
    Authorization: Bearer {your_api_key}
    Since API keys grant broad access, treat them like passwords: store them securely (e.g., in environment variables or a secrets manager, not hard-coded), and transmit only over HTTPS. You can also restrict their usage (for example, some keys can be set to read-only if needed).

All API calls **must** be made over HTTPS. The server will reject any request made over an insecure channel to protect data in transit. Ensure your HTTP client is configured to follow redirects and use TLS 1.2 or above.

For write operations (POST, PUT, DELETE requests), we implement CSRF protection when calls originate from a web browser session. However, in most integration cases you‚Äôll be calling the API from a backend service or script, in which CSRF doesn‚Äôt apply. Your main responsibility is protecting the auth token or API key and including it correctly.

Each API request will be evaluated for authorization: the token or key provided must belong to a user or account with rights to the resource it‚Äôs trying to access. For example, if a token from Company A tries to fetch a report belonging to Company B, the API will respond with HTTP 403 Forbidden. This multi-tenant security model ensures data isolation between organizations using RefCheck.

Response Codes: On authentication failure (e.g., missing/invalid token), the API returns a 401 Unauthorized. If the token is valid but not allowed to access a resource, 403 Forbidden is returned. Expired tokens also lead to 401, so be prepared to refresh tokens if using OAuth. Successful calls typically return 200 (or 201 Created for creation operations). We‚Äôll note specific codes in the endpoint descriptions below.

REST API Endpoints

Key operations for reference checking

Below are the primary endpoints of the RefCheck API, along with explanations and example requests/responses. Using these, you can manage the entire reference check lifecycle programmatically:

  • Create a Reference Check: Start a new reference checking process for a candidate by providing their details and references. This triggers RefCheck to send out questionnaires to those references.
  • Submit Reference Feedback: (Usually done by the reference via the secure link, but can also be done via API for testing or if building a custom reference input form.) Each reference has a unique URL or token to submit their answers, which can be emulated via API.
  • Retrieve Results: Once references respond, you can fetch the compiled report or interim raw answers via API. This allows your system to know when all feedback is in and to access the content for display or further processing.

All endpoints live under the /api/v1/ prefix (assuming the base URL as described). Data is sent and received as JSON. You should include Content-Type: application/json in the header for any request with a body. The examples below illustrate typical usage.

Create Reference Check (POST /reference-checks)

Use this endpoint to initiate a new reference check for a candidate. This typically happens when a candidate reaches the stage where you want to collect references. You‚Äôll provide candidate information, their reference contacts, and optionally specify which question template to use. The response will confirm creation and return an ID for tracking.

i
Note: Before calling this, ensure the candidate has consented to reaching out to these references, as per privacy regulations. RefCheck does not automatically notify candidates; managing consent is up to your workflow.

Request

Example: Create Reference CheckHTTP
POST /api/v1/reference-checks HTTP/1.1
Host: api.refcheck.com
Authorization: Bearer <api_token>
Content-Type: application/json

{
  "candidate": {
    "name": "Alice Johnson",
    "role": "Customer Support Lead"
  },
  "references": [
    { "name": "Bob Manager", "email": "bob@example.com", "relation": "Former Supervisor" },
    { "name": "Carol Coworker", "email": "carol@example.com", "relation": "Colleague" }
  ],
  "template_id": "template_standard_01",
  "callback_url": "https://yourats.com/webhooks/refcheck-done"
}

In this example, we create a reference check for candidate Alice Johnson, applying for Customer Support Lead. We provide two references (Bob and Carol) with their emails and relationship to Alice. The template_id specifies which set of questions to use (here we use a standard template). The callback_url is optional ‚Äì if provided, RefCheck will POST to this URL when the reference check is complete (useful for webhooks, described later).

The API will immediately send out emails (and/or SMS if enabled) to the listed references with secure links for the questionnaire. It will also schedule automatic reminders as configured. The request should include all intended references; if you need to add another later, you would call a different endpoint (or re-invoke this one with an update parameter, if supported).

Response

If successful, the API returns 201 Created along with a JSON body containing the new reference check‚Äôs details. For example:

Response BodyJSON
{
  "reference_check_id": "rc_123456",
  "candidate": {
    "name": "Alice Johnson",
    "role": "Customer Support Lead"
  },
  "references": [
    { "name": "Bob Manager", "email": "bob@example.com", "status": "invited" },
    { "name": "Carol Coworker", "email": "carol@example.com", "status": "invited" }
  ],
  "template_id": "template_standard_01",
  "created_at": "2025-08-01T14:30:00Z",
  "status": "in_progress"

The response echoes back the key information, including a generated reference_check_id that you‚Äôll use to query status or results later. Each reference in the array now has a status (initially ‚Äúinvited‚Äù). As references respond, their statuses will change (e.g., to ‚Äúcompleted‚Äù), which you can fetch via the GET status endpoint.

If the request is malformed or missing required fields, you‚Äôll get a 400 Bad Request with an error message. For example, if an email address is badly formatted or the template_id is invalid, the error JSON will describe the issue. In case of an authentication problem (e.g., missing or wrong token), expect a 401.

Submit Reference Feedback (POST /references/{token}/responses)

This endpoint allows submitting answers for a reference questionnaire. In practice, references fill out the form via the link they receive (which is an interactive form backing onto this endpoint). However, for testing or if building a custom interface, you can POST answers directly. Each reference entry has a unique token (or embedded in their link URL) that identifies the reference and associated candidate.

Endpoint: POST /api/v1/references/{reference_token}/responses

The request body should contain the answers, usually mapped by question ID or text. If using our standard templates, you could send something like:

Example: Submit AnswersJSON
{
  "responses": [
    { "question_id": 1, "answer": "I was Alice's direct manager at XYZ Corp for 2 years." },
    { "question_id": 2, "answer": "Alice's strengths are customer empathy, reliability, and teamwork." },
    { "question_id": 3, "answer": "She could improve technical product knowledge, but she's a quick learner." },
    { "question_id": 4, "answer": 5 },
    { "question_id": 5, "answer": "Yes, definitely" }
  ]
}

When the actual web form is used, the front-end takes each answer the reference provides and calls this endpoint (often question-by-question or all at once at the end). The backend will record each answer in the database. If you call this endpoint directly and send all answers in one payload as above, it will save them in one go.

**Security:** The {reference_token} in the URL is a secret, one-time token. It‚Äôs embedded in the unique link emailed to the reference (typically something like https://app.refcheck.com/ref?token=abc123...). This token cannot be guessed and is tied to that specific reference‚Äôs email. As such, no additional auth header is needed for this endpoint ‚Äì the token in the URL itself authenticates the submission. However, if the token is invalid or expired (e.g., if feedback was already submitted), the endpoint will return 410 Gone or 403 Forbidden accordingly.

Response: On success (answers accepted), a 200 OK is returned, and typically no response body is needed (or just a simple acknowledgment JSON). If you submit answers partially (one by one), you‚Äôll get 200 for each. If the token is wrong or the reference check is closed, you might get a 404 or 410. Validation errors (like missing answers for required questions) yield 400 with details.

Once a reference‚Äôs feedback is fully submitted (all required questions answered and the reference clicked the final submit on the form), RefCheck marks that reference as completed. This triggers any waiting analysis if all references are in, and potentially sends a notification to the recruiter or a webhook to your system. We‚Äôll cover webhooks below.

Endpoint: GET /api/v1/reference-checks/{reference_check_id}

Use the ID returned when you created the reference check. (In our earlier example, it was rc_123456.) The request must be authenticated with a user or API token as it‚Äôs company-specific sensitive data.

Response: On success, returns 200 and a body like:

This output shows that both Bob and Carol have completed their references (with timestamps), and the overall check status is ‚Äúcompleted‚Äù. (The exact fields in report may evolve; always check the latest API docs or include the fields parameter if we support partial responses.) If the check is still in progress, status might be ‚Äúin_progress‚Äù and report could be null or absent. You would then see which references are still ‚Äúpending‚Äù in the list.

Error cases: If you request an ID that doesn‚Äôt exist or belongs to a different company, you‚Äôll get 404 or 403. If the report is not yet ready (e.g., references done but AI analysis pending for a second), the status might still be ‚Äúin_progress‚Äù or ‚Äúanalyzing‚Äù. In such a case, you can try again shortly or rely on the webhook to notify completion.

There is also a possibility to fetch raw reference answers if you need them (e.g., for custom processing). That could be an expanded version of this endpoint or a different one like GET /reference-checks/{id}/responses which lists each question‚Äôs answer by each reference. However, in most cases the summary and flags suffice. If needed, consult the API documentation for the specific resource or include a query parameter to get detailed answers.

Webhooks & Callbacks

Receiving notifications from RefCheck

Constantly polling an API to see if references are done is inefficient. That‚Äôs where **webhooks** come in. RefCheck can send an HTTP POST to a URL of your choosing when certain events occur, essentially ‚Äúpushing‚Äù the update to you. This is especially useful to get immediate notification when a reference check is completed (so you can move the candidate forward, notify a recruiter, etc.).

When creating a reference check via API, you have the option to include a callback_url (as shown in the create example). This URL should be an endpoint in your system that is prepared to receive JSON payloads from RefCheck. If provided, RefCheck will perform an HTTP POST request to that URL under these circumstances:

  • All references completed: As soon as the final reference submits feedback and the report is generated, a webhook is sent. (If the AI report generation is still running, we may send once that‚Äôs done, or send one update for ‚Äúfeedback complete‚Äù and another for ‚Äúreport ready‚Äù. The payload will indicate status.)
  • Reference declined or unreachable (optional): If a reference explicitly declines to give feedback (via an option in the form) or an email bounces, we can notify via webhook. This allows you to take action, like asking the candidate for an alternate reference.

You can also set a default callback_url for all reference checks in your company settings. Then you don‚Äôt need to specify it every time in the API call; RefCheck will use the default unless overridden.

Webhook Request Example

When triggered, RefCheck will POST a JSON payload to your URL. For example:

Webhook PayloadJSON
{
  "event": "reference_check.completed",
  "reference_check_id": "rc_123456",
  "status": "completed",
  "completed_at": "2025-08-03T10:15:30Z",
  "candidate": {
    "name": "Alice Johnson",
    "role": "Customer Support Lead"
  },
  "summary": "Both references praise Alice's reliability and ...",
  "flags": []
}

This payload indicates that the reference check rc_123456 is now complete. We include the summary text and flags directly for convenience (so you don‚Äôt necessarily have to call the API to get it, though you can if you want full detail). The event field tells the type of webhook. Currently, the main event is reference_check.completed but in future, we might add others (like reference.delayed or reference.reminder_sent if those are useful). Your endpoint should parse the JSON and confirm receipt by returning a 200. If we get a non-2xx response or a timeout, RefCheck will retry the webhook a few times (with exponential backoff) to handle transient issues.

Security: To ensure the webhook is coming from RefCheck and not someone else, you can do a couple of things:

  • Include a secret token in the URL that only you and RefCheck know, e.g., https://yourapp.com/refcheck-webhook?token=ABC123. RefCheck will hit that exact URL, and you verify the token on receipt.
  • Alternatively, we can sign the webhook requests (for example, include a signature header using an HMAC and a shared secret). This is a bit more involved but more secure. Check RefCheck settings to see if webhook signing is available; if so, your code would compute a hash of the payload using the secret and compare it to the signature header.

Ensure your webhook endpoint is configured to handle the incoming data quickly. Typically, you might just mark something in your database or enqueue further processing, then return a 200. If your endpoint consistently fails, RefCheck may disable webhooks for your account until the issue is resolved to avoid overwhelming your server with retries.

ATS Integration Example

Let‚Äôs walk through a typical integration with an Applicant Tracking System (ATS) to illustrate how the pieces come together:

üë©‚Äçüíº Scenario: Rachel integrates RefCheck with Greenhouse ATS

Rachel wants any candidate moved to the ‚ÄúReference Check‚Äù stage in Greenhouse to automatically trigger RefCheck, and once references are done, she wants a note added to the candidate‚Äôs profile.

  1. Triggering via Webhook/Zapier: Greenhouse (and many ATS) allow setting up a webhook or using an automation platform like Zapier. Rachel configures a webhook in Greenhouse: whenever a candidate‚Äôs stage changes to ‚ÄúReferences,‚Äù an HTTPS POST is sent to a specific URL. She sets that URL to a small integration service (or a Zapier webhook trigger) that she controls.
  2. Calling RefCheck API: Upon receiving the ATS webhook with candidate details (name, email, job, etc.), Rachel‚Äôs integration service calls the RefCheck API (the ‚ÄúCreate Reference Check‚Äù endpoint). It plugs in the candidate info and finds the references either from the ATS data or by looking them up in a separate form. Alternatively, if Greenhouse stores references in custom fields on the candidate, the webhook might include them, which can be passed through. This API call returns the new reference_check_id, which the service can store along with the ATS candidate ID for correlation.
  3. RefCheck in action: RefCheck sends out emails to the references. Rachel can monitor progress in the RefCheck dashboard, but she doesn‚Äôt need to do anything manually ‚Äì it‚Äôs all running in parallel to her ATS.
  4. Completion Webhook back to ATS: Rachel provided a callback_url in the API call (or set a default one) that points to another endpoint in her integration service. When RefCheck posts the ‚Äúcompleted‚Äù event, her service receives it, parses the summary and outcome.
  5. Updating ATS profile: Using the Greenhouse API, the integration service posts a note to the candidate‚Äôs profile: e.g., ‚ÄúReferences completed on Aug 3. Summary: Alice is highly recommended by former manager and colleague. No red flags.‚Äù It might also move the candidate to the next stage (e.g., ‚ÄúReady for Offer‚Äù) by calling another ATS API endpoint, or at least notify the hiring team via email/Slack.

In this scenario, Rachel doesn‚Äôt have to do any data entry in RefCheck; everything is driven by ATS events. The key integration points are the outgoing webhook from ATS to trigger the process, the API call to RefCheck to create the check, and the webhook from RefCheck to notify completion. Each system remains authoritative for its part (ATS for candidate info, RefCheck for reference processing), but they communicate seamlessly.

This model can be adapted to other ATS or HR systems. Some systems might not have webhooks but allow scheduled polling ‚Äì in that case, you could periodically call the ATS API to find candidates in ‚ÄúReference Check‚Äù stage and then call RefCheck accordingly. However, webhooks are preferred for real-time and efficiency.

Zapier & Other Tools

If you don‚Äôt want to write a custom integration service, or your team prefers no-code solutions, RefCheck can work with automation platforms like **Zapier**, **Make (Integromat)**, or others:

For instance, you can create a Zap: *Trigger*: ‚ÄúNew Stage Change in [Your ATS] (stage = Reference Check)‚Äù ‚Üí *Action*: ‚ÄúHTTPRequest (POST) to RefCheck API to create reference check‚Äù. Additionally, you can have another Zap where *Trigger*: ‚ÄúWebhook from RefCheck (Reference Check Complete)‚Äù ‚Üí *Action*: ‚ÄúUpdate Candidate in ATS (add note or move stage)‚Äù. We provide templates in Zapier to simplify this ‚Äì essentially, those templates know which fields to map (candidate name, email, etc.). Using Zapier is great for initial automation or if you lack developer resources. Just be mindful of Zapier‚Äôs rate limits and ensure you don‚Äôt expose the API keys publicly within Zap configurations.
  • Native Integrations & Plugins: Over time, RefCheck aims to offer native integrations (plugins) for popular systems. For example, a direct Greenhouse plugin might let you trigger reference checks with one click inside Greenhouse‚Äôs UI, without even dealing with API keys.   • Other HRIS/HCM Systems: You might want to integrate reference check outcomes into onboarding software or HRIS. Remember, when using third-party automation tools, treat your API credentials carefully. Do not share API keys in channels or code repositories. In Zapier, for instance, use their secure authentication or store the key in a Zapier secret field. Consider creating a special Integration User in RefCheck with limited access (if that capability exists) so that the API key used cannot, say, delete data or access other areas beyond what‚Äôs needed.

Ultimately, our API and webhook flexibility means you can integrate RefCheck with just about anything ‚Äì whether it‚Äôs a modern ATS, a legacy system, or even a Slack workflow (imagine a Slack bot that posts ‚ÄúReference check complete for Alice, type /view_ref Alice to see results‚Äù ‚Äì you could build that using the API!). We encourage experimentation and provide support for development questions.

Data Schema & Examples

Understanding templates and responses

The RefCheck system deals with structured data for questionnaires and responses. While the API abstracts most of this (you don‚Äôt usually need to know the internal schema to use it), it can be helpful to understand how things are structured ‚Äì especially if you plan to customize questions or interpret detailed results. Here are a couple of important data structures:

Questionnaire Template

A template defines the questions asked to references. Each reference check uses one template. By default, we have a ‚ÄúStandard 5-question Template‚Äù (referenced by template_id like template_standard_01). If your organization needs different questions for certain roles, you can define multiple templates (currently via the RefCheck UI or by contacting support; in future, via API). The template schema looks like:

Questionnaire Template SchemaJSON
{
  "template_id": "template_standard_01",
  "name": "Standard Reference Template",
  "questions": [
    { "id": 1, "text": "How do you know the candidate and in what context?", "type": "text" },
    { "id": 2, "text": "What are the candidate's greatest strengths?", "type": "text" },
    { "id": 3, "text": "What areas could they improve upon?", "type": "text" },
    { "id": 4, "text": "Rate the candidate's reliability from 1-5.", "type": "rating", "scale": 5 },
    { "id": 5, "text": "Would you rehire the candidate?", "type": "multiple_choice",
      "options": ["Yes, definitely", "Possibly / with reservations", "No"] }
  ]
}

This JSON isn‚Äôt something you usually pull via API (though we may add an endpoint to list templates). It‚Äôs shown to clarify the question_id references you see in responses. Each question has an ID and type. Types can be text (open-ended), rating (numeric scales), multiple_choice, etc. When references answer:

  • Text and multiple-choice answers come back as strings (the exact text or the chosen option text).
  • Rating answers come back as numbers (as seen in the example for question 4 above, answered with 5).
If you had a custom template with different questions, the response payloads would use those question IDs. It‚Äôs crucial that if you do vary templates, your integration code accounts for that when parsing responses or reports (for example, a sales position might have an extra question about meeting targets).

Raw Responses

While the summary report is usually enough, you might have a use case to store or analyze the raw answers. We haven‚Äôt made a dedicated endpoint for raw answers in v1, but the data is accessible in the reference check status (if requested with a detail flag, or a separate endpoint can be enabled). A raw response structure per reference might look like:

Reference Responses (Sample)JSON
{
  "reference_id": "ref_101",
  "name": "Bob Manager",
  "responses": [
    { "question_id": 1, "answer": "I supervised Alice at XYZ Corp from 2018-2020." },
    { "question_id": 2, "answer": "She has great customer empathy and is very reliable." },
    { "question_id": 3, "answer": "She could improve her technical product knowledge." },
    { "question_id": 4, "answer": 5 },
    { "question_id": 5, "answer": "Yes, without hesitation." }
  ]
}

This example shows the kind of detail that underlies the summary. If needed, we can provide these via an export or API. Many integrations won‚Äôt need to go to this level, but if you were doing data analysis (e.g., tracking common weaknesses across all hires) or feeding data into a machine learning system, you‚Äôd use these. Keep in mind to respect privacy ‚Äì these free-text answers may contain personal opinions and should usually remain internal.

One more note: Each reference in the system has its own unique ID (reference_id like ref_101 above) ‚Äì different from the reference check id. Under the hood, this ties the reference‚Äôs responses and contact info together. When you use the reference check status endpoint, we can include an array of reference objects with their response completion as shown earlier, but not necessarily all their answers unless requested.

Appendix: Sentiment Analysis Example

How RefCheck flags answers (developer insight)

This section provides a peek under the hood at how RefCheck‚Äôs AI might perform sentiment analysis on reference feedback. You generally do not need this to use the API, but it can help interpret the flags or sentiment_score you see in reports. The actual AI uses a complex model, but here‚Äôs a simplified pseudocode that captures the essence of flagging positive/negative language in a reference‚Äôs answer:

Sentiment Analysis PseudocodeJS
function analyzeFeedback(text) {
  // Basic sentiment detection using keyword matching (fallback if NLP fails)
  const positiveHints = ["always", "excellent", "great", "definitely", "strongly recommend"];
  const negativeHints = ["but", "however", "concerns", "would not", "reservation"];
  let score = 0;
  for (const hint of positiveHints) {
    if (text.toLowerCase().includes(hint)) score += 1;
  }
  for (const hint of negativeHints) {
    if (text.toLowerCase().includes(hint)) score -= 2;
  }
  return score;
}

// Example usage:
const feedback = "She was always reliable and exceeded expectations, but needed some guidance on technical tasks.";
console.log(analyzeFeedback(feedback));  // score -> -1 (mixed sentiment)

In this snippet, certain words adjust a sentiment score. Highly positive words increment the score, whereas words that often indicate a caveat or issue (like ‚Äúbut‚Äù or ‚Äúconcerns‚Äù) decrement more heavily. In the example feedback, the score comes out negative due to the ‚Äúbut needed some guidance‚Äù clause, even though positive words were present ‚Äì hence a mixed or slightly cautionary sentiment.

RefCheck‚Äôs AI does much more than this, including understanding context and synonyms, but the idea is similar: it tallies evidence of praise versus caution. The flags array you see in a report might highlight exact phrases like ‚Äúneeded some guidance on technical tasks‚Äù as a yellow flag for improvement area. A red flag might be something like ‚Äúwould not rehire‚Äù or ‚Äúattendance issues,‚Äù etc. These help the recruiter quickly spot potentially problematic feedback without combing through all text.
