Design Overview

A user-centric interface for fast, intuitive reference checks

RefCheck‚Äôs UI is designed to be clear, modern, and extremely easy to use for all parties involved. There are two primary user groups: internal **recruiters/HR** who manage reference checks, and external **reference providers** who give feedback. The interface caters to both: recruiters get a clean dashboard to launch and track checks with minimal clicks, and references experience a friendly, guided **chat-style survey** that feels conversational rather than tedious. Overall, the design emphasizes **speed** (completing references in 24 hours or less, with each referee spending ~5 minutes) and **clarity** (each step is obvious and requires little to no training).

The application uses a **conversational design paradigm** where appropriate (for the reference questionnaire) to put users at ease and encourage candid input. At the same time, it maintains a professional and trustworthy look aligned with HR expectations (e.g. clean layouts, company branding present) so that both candidates and references feel confident in the process. All interactions ‚Äì from sending reference invites to reviewing a final report ‚Äì happen through a cohesive design language, ensuring a smooth experience without confusion or unnecessary complexity. In short, the UI/UX aims to turn a traditionally painful process into a seamless, even pleasant, workflow.

Design Principles & Goals

Core values guiding the user experience

  • Ease of Use: Every interaction should feel intuitive. Recruiters can initiate and monitor checks without extensive training, and references can complete feedback with zero confusion. The UI avoids jargon and uses clear labels (e.g., ‚ÄúSend Reference Request‚Äù or ‚ÄúSubmit Feedback‚Äù).
  • Efficiency: Speed is a priority ‚Äì minimizing the time and steps needed to accomplish tasks. For recruiters, sending out a reference questionnaire takes just a few fields and a click. For references, the chat interface streamlines giving feedback by prompting one question at a time and guiding them promptly through completion.
  • Conversational & Engaging: Especially for reference providers, the experience is designed to feel like a quick conversation rather than a formal survey. This principle increases engagement and candor. Friendly microcopy (e.g., the assistant says ‚ÄúHi and thanks for helping us!‚Äù) and a chat bubble interface help put users at ease.
  • Clarity & Transparency: The design presents information in a structured and predictable way. Progress indicators (e.g., ‚ÄúQuestion 3 of 5‚Äù) ensure users know how much is left. Important messages (such as confidentiality notices or error hints) are prominently displayed. The report for recruiters clearly highlights key insights without burying information.
  • Consistency & Professionalism: All components of RefCheck share a consistent visual style and tone, reinforcing user confidence. Colors, typography, and spacing are uniform across the dashboard, emails, and forms. The UI integrates the hiring company‚Äôs branding (logo and name) in a tasteful way, giving a professional impression that reflects well on the company using RefCheck.
  • Inclusivity & Accessibility: The product is designed to be usable by a wide range of people, including those on mobile devices or with assistive technologies. Sufficient contrast, legible font sizes, and support for screen readers and keyboard navigation are built in (detailed in the Accessibility section). No user should be excluded or frustrated due to design choices.

Branding & Visual Style

Look and feel consistent with company branding and user trust

RefCheck‚Äôs visual design is clean, modern, and neutral enough to blend with a client company‚Äôs branding, while still providing a distinct, friendly interface. Key aspects of the visual style include:

Color Palette

The interface uses a balanced color scheme built around a primary blue and complementary accents:

  • Primary Blue (#0B63CE): Used for primary actions (e.g., buttons like ‚ÄúSend Invite‚Äù or ‚ÄúSubmit‚Äù), highlights, and active states. This blue conveys trust and professionalism, aligning with typical corporate branding.
  • Secondary Green (#00A36C): A secondary accent appearing in the brand gradient and success states. It‚Äôs used subtly ‚Äì for example, in the header background gradient and to indicate completion or success (a checkmark icon might be green, success alerts have a green left border).
  • Accent Purple (#7C3AED): A tertiary color used sparingly to draw attention to special elements (for instance, the ‚ÄúPowered by RefCheck‚Äù tag or AI-related highlights). It provides a touch of vibrancy without overwhelming the primary blue/green theme.
  • Neutral Palette: Generous white and light grey backgrounds (e.g., #F6F6F7 for page background, #FFFFFF for cards and form surfaces) with mid-grey text (#4B5563) ensure readability. Borders and separators use light grey (#E5E7EB) for a subtle delineation of sections and fields.
  • By default, the app‚Äôs colors suffice for most companies, but in **white-label** scenarios, these can be customized. Company logos are displayed on the reference invitation and form, and colors are chosen to complement rather than clash with clients‚Äô branding.

Typography & Iconography

The app uses a sans-serif font stack (system fonts like Segoe UI, Roboto, etc.), providing a clean, familiar look. Text hierarchy is achieved through size and weight rather than multiple font families. For example, section headers and card titles are slightly larger and bolder, while body text remains at a comfortable base size (~14-16px). All text is dark grey on light backgrounds for maximum legibility.

Icons are used sparingly and intentionally to reinforce meaning:

  • Status Icons: Simple icons accompany statuses and flags. For instance, a green checkmark (‚úì) denotes completed or positive feedback, a yellow caution symbol (‚ö†) flags a potential concern, and a red stop icon (‚õî) marks a serious issue. These icons appear next to text labels or in tooltips, never on their own, so that color-blind users or those unfamiliar with the icon still have context from text.
  • UI Icons: Common actions have intuitive icons (e.g., a ‚Äú+‚Äù plus icon on the ‚ÄúNew Reference Check‚Äù button, a filter icon for filtering lists, a download icon for ‚ÄúExport PDF‚Äù). We leverage a standard icon set (such as FontAwesome or similar) for consistency. Icons are kept moderate in size and neutral in color (using the text color) unless emphasizing an interactive hover state.
  • Illustrations/Logos: The hiring company‚Äôs logo appears in the email header and at the top of the reference form page, scaled appropriately. If the company has no logo provided, a default RefCheck logo or placeholder initials are used. No other heavy graphics or illustrations are used in the workflow, keeping the design lightweight and focused on textual content.

Layout & Spacing

The design employs a card-based layout for grouping information. For example, on the recruiter dashboard, each candidate‚Äôs reference check may appear as a card or row with padded content and a subtle shadow on hover, reinforcing clickability. Forms and dialogs appear in centered panels or modals with rounded corners (12px radius) to match the friendly aesthetic.

Generous spacing is used to avoid a cramped feel: sections are separated by clear margins, and form fields and buttons have sufficient padding to be easily tappable. We maintain approximately 16px margins around most text blocks for readability. Lists and tables have row striping or dividing lines to help scan data.

Overall, the visual style is **consistent**: repeated components (buttons, inputs, alerts) look and behave the same throughout the app. For example, all primary buttons are the same blue with white text and a slight hover darkening; all alerts use the left-border color convention (blue for info, orange for warnings, green for success, red for errors) with matching icons. This consistency means once a user learns a pattern (e.g., a dotted underline indicates hoverable help text), they can predict it elsewhere, reducing cognitive load.

Key Screens & Components

Primary user interfaces and their elements

RefCheck‚Äôs user experience spans a few main screens/components, each optimized for its user‚Äôs needs. Below is an overview of each:

  • Recruiter Dashboard: The central hub for HR users to initiate new reference checks and monitor progress on ongoing ones. It lists all candidates undergoing references, and provides status at a glance (e.g., ‚Äú2 of 3 references completed‚Äù).
  • Reference Invitation Email: An email template that is sent to reference providers, branded with the hiring company‚Äôs identity. It contains a personalized request and a secure link for the reference to start the questionnaire.
  • Reference Chat Form: The conversational reference collection interface on the web. This is where a reference provider answers questions in a chat-like format, one question at a time, with the AI assistant guiding the flow.
  • Report & Analytics: The results view for recruiters once references are in. It includes an AI-generated summary, the question-by-question feedback from all references (with any flags or highlights), and tools to export or share the report.

Recruiter Dashboard UI

The recruiter dashboard is a web interface (desktop-optimized but responsive) that greets the internal user upon login. Its design focuses on giving **immediate insight** and quick actions. Key elements include:

  • Reference Check List: A table or card list of all reference checks in progress and recently completed. Each entry typically shows the candidate‚Äôs name, the job role, date started, and current status. Status is conveyed both in text (e.g., ‚Äú2/3 Complete‚Äù) and color/icon ‚Äì for example, orange text or a half-filled progress bar if some references are pending, turning green with a checkmark when all completed. This lets recruiters scan which candidates need attention.
  • Quick Actions: Each reference check entry provides actions like ‚ÄúView Report‚Äù (opens the detailed report, even if not fully complete yet), and if references are still pending, a ‚ÄúSend Reminder‚Äù action. Icons (such as a paper-plane for reminder) with tooltips make these actions recognizable. The design ensures these action buttons are subtle but discoverable (e.g., appearing on hover or always shown as small icons to the right of each row).
  • New Reference Check Flow: A prominent button labeled **‚ÄúNew Reference Check‚Äù** (often top-right of the dashboard) opens the workflow to create a new check. This could be a modal or separate page with a form (as described in the Setup section of the PRD). The UI guides the recruiter to input candidate name, email, role, and reference contact info, select a question template, etc. The design uses a multi-step form or well-grouped fields so that it doesn‚Äôt appear daunting. For instance, references‚Äô contact fields might have an ‚ÄúAdd another reference‚Äù button to dynamically add fields.
  • Notifications & Alerts: The dashboard surface also serves to notify the recruiter of important events. For example, if a reference responded, a non-intrusive alert (a green toast message at the bottom or a small badge on the candidate‚Äôs entry) indicates ‚ÄúReference from John Doe received.‚Äù Errors, such as an invite email bouncing, are highlighted in context (the affected reference might show an error icon and an option to edit the address). These cues ensure the recruiter doesn‚Äôt miss anything critical upon opening the dashboard.
  • Filtering and Sorting: If the HR team handles many candidates, the UI provides controls to filter or search. A search bar might let them find a candidate by name. Filter dropdowns can narrow the list (e.g., show only ‚ÄúPending‚Äù checks or filter by job role or hiring manager). These controls are placed clearly at the top of the list and use common icons (like a funnel for filter, magnifying glass for search).
  • Collaboration Indicators: In cases of multiple recruiters, the dashboard may indicate ownership or sharing. For example, each check might show the owner‚Äôs initials or name, or a lock icon if only that recruiter can view it. The UI aligns with RBAC rules from the PRD, ensuring users only see what they should. Design-wise, these indicators are subtle (muted text or an icon) to avoid clutter but available on hover for details (‚ÄúCreated by Rachel, last updated 2h ago‚Äù).

Overall, the dashboard‚Äôs design balances **overview and control**. The top of the dashboard could also include a summary bar (especially for high-volume users) with KPIs like ‚ÄúTotal Checks: 15 | Pending References: 4 | Completed Today: 3‚Äù, giving a quick health check of their pipeline. Everything on this screen is laid out to minimize the time a recruiter spends ‚Äì they can get in, see what needs action, take that action, or quickly launch a new check, all in a few clicks.

Reference Invitation Email

This is the first touchpoint for a reference provider. The email‚Äôs design is simple, email-client friendly, and professional, with a touch of the company‚Äôs branding. Key characteristics:

  • Company Branding: The email uses the hiring company‚Äôs name and logo at the top of the message. Typically, the company logo appears as a small header image (left-aligned), and the text ‚Äú[Company Name] Reference Request‚Äù is in the subject line and header, immediately letting the recipient know the context. The color scheme of the email sticks to neutral white background with black/gray text, and uses the company‚Äôs primary color or the RefCheck blue for the call-to-action button.
  • Personalized Greeting: It opens with a greeting to the reference by name (e.g., ‚ÄúDear John,‚Äù or ‚ÄúHi John,‚Äù) to set a friendly tone. The first line establishes the connection: e.g., ‚Äú[Candidate Name] has listed you as a reference for the [Job Title] position at [Company Name].‚Äù This is in plain text (no images), ensuring it displays on all email clients.
  • Concise Message: The body text briefly thanks the reference for their time and explains what is being asked. For example: ‚ÄúWe‚Äôd greatly appreciate it if you could answer a short, 5-minute questionnaire about your experience working with [Candidate]. Your insights will help us in our hiring decision.‚Äù It also reassures about confidentiality (‚ÄúYour feedback will be kept strictly confidential and only used by our hiring team.‚Äù). The tone is polite and appreciative.
  • Clear Call-to-Action: A prominent button with wording like **‚ÄúProvide Reference‚Äù** or **‚ÄúStart Questionnaire‚Äù** is centered in the email. This button is typically in the primary brand color (blue) with white text, and has a decent touch target (bulletproof HTML/CSS for broad client compatibility). It links to the unique, secure survey URL. For clients that block images or styles, a fallback plain-text link is provided below the button (‚ÄúOr paste this URL into your browser: [URL]‚Äù).
  • Footer Information: The email closes with a polite thank-you (‚ÄúThank you for your time and help in our hiring process!‚Äù) and the sender‚Äôs identity. Usually this includes the name and title of the recruiter or hiring manager (e.g., ‚ÄúSincerely, Rachel Adams, Talent Acquisition Manager, [Company]‚Äù). There‚Äôs also a line with contact info in case the reference has questions or issues (e.g., a reply-to email or phone number for HR). If required by compliance, an unsubscribe link or notice is included (though, since this email is transactional and expected, the tone is that this is a one-time request, not marketing).
  • Mobile-Friendly: The email is designed in a single-column layout and tested on mobile devices. Fonts are a readable size (at least ~16px for body text on mobile), and the CTA button is large enough to tap. Any padding or margins ensure the content isn‚Äôt flush to the screen edges on a phone. Images (like the logo) have appropriate alt text (‚Äú[Company Name] logo‚Äù).

The overall appearance is **professional and uncluttered** ‚Äì mostly text on a white background, with one accent color used for the response button. This simplicity ensures the email loads quickly and looks consistent across email clients (Outlook, Gmail, mobile email apps, etc.). It also reduces the chance of being flagged as spam: the email is clearly a personal request, not a mass marketing mail, and the wording reflects that.

Reference Chat Form UI

The Reference Chat Form is the heart of the reference provider experience ‚Äì a web page (mobile-responsive) where they answer the questionnaire. It‚Äôs designed to feel like a simple chat conversation with an assistant, leveraging familiar messaging app patterns. Key design and UX elements:

  • Welcome Screen: When the reference clicks the invite link, they arrive at a welcome page that‚Äôs branded with the company logo and colors. A brief title might say ‚ÄúProvide a Reference for [Candidate Name]‚Äù and include a one-line reassurance like ‚ÄúThis will only take about 5 minutes.‚Äù A **‚ÄúStart‚Äù** button or the first chat message prompt (‚ÄúHi [Reference Name]! ...‚Äù) invites them to begin. This page ensures the reference knows who and what it‚Äôs for before diving in.
  • Chat Interface Layout: Once started, the screen resembles a chat window:
    • The **AI Assistant‚Äôs messages** (questions) appear as speech bubbles aligned to the left. They might have a subtle background color (light grey or a faint blue) and possibly an avatar icon (for example, the company‚Äôs logo or a generic chat icon) to indicate ‚Äúthis is the system speaking.‚Äù
    • The **reference‚Äôs responses** appear as bubbles aligned to the right (similar to how personal messages appear in iMessage/WhatsApp), in a contrasting shade (like company-themed light blue or green) to distinguish them. This left-right pattern visually separates questions and answers clearly.
    After each answer is submitted, the reference sees their text in a bubble, then the next question bubble appears.
  • One Question at a Time: The form delivers questions sequentially to avoid overwhelming the user. Only one (or the current) question input is active at any time, usually displayed at the bottom of the chat window. The design uses a fixed position input field:
    • For open-ended questions: a simple text box appears with placeholder text like ‚ÄúType your answer...‚Äù and a **Send** button (often an arrow icon) to submit. Hitting Enter/Return also sends the answer, mimicking chat behavior.
    • For multiple-choice or ratings: the UI presents those options as tappable chips or buttons above the text input. For example, a ‚ÄúRate reliability 1-5‚Äù question might show five star icons or five buttons labeled 1 through 5 that the user can simply tap. If ‚ÄúWould you rehire?‚Äù is a multiple choice (Yes / With reservations / No), those three options could appear as selectable buttons. When selected, the choice is displayed as the user‚Äôs ‚Äúsent‚Äù message.
    This adaptive input method reduces effort ‚Äì references don‚Äôt need to type when a simple selection suffices ‚Äì and it ensures structured answers.
  • Guidance & Probing: The conversational UI allows dynamic prompting. The assistant may post a follow-up question in the flow if needed. Design-wise, follow-ups appear as just another left-aligned bubble, often with a slight indent or a connecting line to indicate it‚Äôs linked to the previous question‚Äôs answer. For example, if the reference mentions ‚ÄúHe needed to improve technical knowledge,‚Äù the assistant might respond with a bubble: ‚ÄúThanks. Can you elaborate on the technical areas he needed help with?‚Äù This immediate branching keeps the flow natural. If the reference chooses not to elaborate, they can simply type a short answer or even ‚ÄúN/A‚Äù and move on.
  • Progress Indicator: To manage expectations, the interface shows progress. This can be a small indicator at the top of the chat or in the header (‚ÄúQuestion 2 of 5‚Äù) or a progress bar that fills as they answer questions. It provides context so the reference knows roughly how many questions remain, reducing drop-offs. The style is unobtrusive ‚Äì e.g., a light progress bar below the header or a subtle text that updates with each question.
  • Skip & Save Options: Each question includes a way to skip if the reference truly cannot answer. This might be a **‚ÄúSkip question‚Äù** link or button near the text input. Skipped questions might trigger a bubble that says ‚Äú(Reference chose to skip this question)‚Äù or the assistant bubble might simply say ‚ÄúNo problem, we‚Äôll skip that one.‚Äù Additionally, if the user closes the form mid-way, the system saves their progress (answers submitted so far). When they reopen the same link, the chat can either resume where they left off or show a message like ‚ÄúWelcome back, shall we continue where you left off?‚Äù This ensures even busy references can come back without losing data. (From a design perspective, a small note might be shown like ‚ÄúProgress saved automatically.‚Äù)
  • Review & Edit Answers: After the last question, the assistant presents a **summary of all answers**. This appears as a list within the chat or a summary screen listing each question with the reference‚Äôs response. The design uses a distinct style for this summary ‚Äì e.g., a card or simply a series of bubbles from the assistant summarizing: ‚ÄúHere‚Äôs what we‚Äôve got...‚Äù followed by each Q&A pair. Each answer may have an **‚ÄúEdit‚Äù** option (like a small edit icon or an ‚ÄúEdit‚Äù link next to it) that, when clicked, allows the reference to modify their response to that question (either by scrolling the chat back to that position or by reopening the input for that question in place). This edit flow is made as simple as possible: the user doesn‚Äôt have to re-run the entire chat, just adjust the specific answer and reconfirm.
  • Submission & Confirmation: At the end of the summary review, a prominent **‚ÄúSubmit Reference‚Äù** button is provided. The reference must click this to finalize their feedback. Once submitted, the UI immediately displays a thank-you message: for example, a friendly illustrated icon (like a thumbs-up or checkmark) and text: ‚ÄúThank you! Your reference for [Candidate] has been submitted.‚Äù The page also might note, ‚ÄúWe appreciate your time. You may close this window.‚Äù Optionally, as per PRD, a quick satisfaction survey is shown: a row of 5 stars asking ‚ÄúHow was your experience providing this reference?‚Äù The reference can rate it, and perhaps a textbox for additional comments (this is for our internal improvement, not seen by the recruiter). This feedback UI is minimal and skippable.

Throughout the chat form, the design priorities are **simplicity and mobile usability**. The text is kept concise and friendly. On mobile, the chat bubbles fill most of the screen width, with ample padding for touch. On desktop, the chat is centered in a narrower container so it still feels intimate (often ~400px width like a chat app). There are no intrusive elements ‚Äì no ads, no complex menus ‚Äì just a straightforward flow. If at any point the reference encounters an error (e.g., their internet drops on submission), the UI shows a clear error state message in the chat (e.g., ‚ÄúNetwork issue ‚Äì please check your connection and press Submit again‚Äù) with a retry option, ensuring they can complete the process.

Report & Analytics View

Once all references for a candidate are in (or even as they come in, for interim viewing), the recruiter can view the compiled report. The design of this report focuses on clarity, compare-ability of responses, and highlighting of key insights. The Report view typically appears either as a dedicated page or a modal/dialog from the dashboard when ‚ÄúView Report‚Äù is clicked. Its design includes:

  • Header Summary: At the top, the AI-generated summary of the candidate‚Äôs references is prominently displayed. This might be styled as a block quote or a distinct highlighted section. For example, it could be a slightly shaded background with an ‚ÄúAI Summary‚Äù label, containing a few sentences summarizing strengths and weaknesses. Alongside this summary, there may be quick stats: e.g., a count of references received vs requested, and overall sentiment indicators. If an overall ‚Äúreference score‚Äù or rating is calculated (though we avoid an explicit score, we might show something like a sentiment meter), it could be displayed here. For instance, a horizontal bar or icon indicating ‚ÄúOverall Sentiment: Positive‚Äù with a green dot. If any reference expressed serious concerns (flags), a small alert icon might appear near the summary with a tooltip like ‚ÄúSome cautionary feedback included.‚Äù
  • Reference Details by Question: Below the summary, the feedback is structured question by question. Each question from the template is listed, with each reference‚Äôs answer shown for that question. There are a couple of possible layouts:
    • Tabular layout: A table where each row is a question and each column is one reference‚Äôs answer, labeled by the reference‚Äôs name or role. This allows side-by-side comparison of how different references answered the same question. If there are many references (up to 6), the table can scroll horizontally on smaller screens or stack rows vertically on mobile (question followed by answers).
    • Stacked Q&A layout: Each question is a section heading, and beneath it, each reference‚Äôs answer is shown as a paragraph or bulleted item, prefaced with the reference‚Äôs name/relationship (e.g., ‚ÄúJohn Doe (Former Manager): [His answer].‚Äù). This is more vertical but ensures readability, especially if answers are long.
    We often use a hybrid: on wider screens, answers render in a grid or table; on narrow screens, they stack. The UI uses subtle backgrounds or border lines to separate each question section.
  • Highlighting & Flags: The report automatically emphasizes important points:
    • Any **flagged phrases or concerns** identified by AI are highlighted (e.g., yellow highlight on the text, or a caution icon right next to the sentence). For instance, if a reference wrote ‚Äúsometimes late to meetings,‚Äù that phrase might be underlined in orange with a ‚Äú‚ö†‚Äù icon. If the recruiter hovers or taps it, a tooltip might say ‚ÄúThis was flagged as a potential concern.‚Äù This draws the recruiter‚Äôs eye to areas needing attention.
    • Sentiment analysis results can be indicated by color or icons near each answer or reference name. For example, next to each reference‚Äôs name we might show a small face icon: a green smiling face for predominantly positive feedback, a neutral face for mixed, etc., based on the sentiment score. This provides an at-a-glance sense of each reference‚Äôs tone without having to read every word.
    Any critical red flags (like a ‚Äúwould not rehire‚Äù response) would be very clearly marked ‚Äì perhaps the text itself in a red font or an urgent icon next to the reference‚Äôs name in the summary section (‚ÄúReference Mary indicated she would not rehire the candidate ‚õî‚Äù).
  • Follow-up Suggestions: Leveraging the AI analysis, the UI may present a section for ‚ÄúSuggested Follow-ups‚Äù at the bottom of the report or inline with relevant answers. This could be styled as an info alert box or italicized text. For example: ‚ÄúFollow-up suggestion: Ask the candidate about their experience improving technical skills, since one reference noted initial struggles in that area.‚Äù These suggestions are clearly distinguished (perhaps with a lightbulb icon or a different font style) so that the recruiter knows this is guidance, not reference content.
  • Collapsible/Expandable Sections: If some questions elicited very long answers, the UI might collapse them by default for readability, especially if viewing summary on a single page. Each answer could show the first line or two with a ‚ÄúRead more‚Äù expand option. This way, the recruiter can skim through quickly and only delve into the full text of interest. Alternatively, a ‚ÄúExpand All‚Äù toggle could let the recruiter switch between a concise view and a full view.
  • Export & Share Controls: At the top or bottom of the report view, there are actions for the recruiter:
    • **Download PDF:** A button (with a PDF or download icon) to export the entire report in a nicely formatted PDF. The PDF mirrors what‚Äôs on screen (the summary, Q&A, etc.), using the same styling as much as possible, adjusted for print (company logo, candidate name and date on a cover page, etc.).
    • **Share Link:** If the recruiter wants to share the report with a hiring manager or others, the UI provides either a ‚ÄúCopy link‚Äù button or an ‚ÄúAdd viewer‚Äù function. For simplicity, a ‚ÄúCopy Secure Link‚Äù could generate a unique URL that can be sent to others (as mentioned in the PRD scenario). The UI will indicate any access limitations (e.g., ‚ÄúAnyone with this link can view the reference report ‚Äì expires in 7 days‚Äù to maintain confidentiality). This control is placed near the top, with a tooltip explaining it.
    • Possibly an **‚ÄúEdit References‚Äù** action if any reference feedback was taken by phone and needs to be entered manually (though typically all feedback comes through the form, an admin might have this option). This would open an interface to input or adjust data, but such an option might be hidden for normal users to prevent tampering.

The visual style of the report prioritizes readability: plenty of whitespace, a clear different font or style for questions vs answers (e.g., questions in bold, answers in normal text). If using a table, alternating row colors or separators help distinguish between questions. The design is **print-friendly** ‚Äì background colors and images are minimized so that when printed, it looks clean (the PDF export handles this primarily).

Finally, the report page may include navigation aids if accessed via the dashboard (like a ‚ÄúBack to Dashboard‚Äù link or breadcrumb at top). After reviewing, the recruiter can easily return to the candidate list. The overall UX ensures the recruiter can quickly extract insights ‚Äì the AI summary and highlights do the heavy lifting, and the structured layout allows easy verification by reading actual comments.

Accessibility & Responsive Design

Inclusive design for all users and devices

RefCheck is built to be usable by anyone, regardless of device or ability. We adhere to accessibility best practices (aiming for WCAG 2.1 AA compliance) and ensure the UI works well on various screen sizes. Key considerations include:

  • Responsive Layout: Every page and component adjusts to different screen widths. On small mobile screens, layouts switch to single-column, and text and buttons enlarge for easy tapping. For instance, the recruiter dashboard collapses any multi-column tables into stackable cards (each showing candidate info and status vertically) or introduces horizontal scroll for overflow. The reference chat form was designed ‚Äúmobile-first‚Äù ‚Äì it occupies the full screen on a phone, with chat bubbles and input sized for touch. No element should require horizontal scrolling on a narrow device (except where data tables must scroll intentionally).
  • Scalable Text: Users can zoom in (increase font size) without breaking the layout. We use relative font sizes (e.g., em, rem) so that if someone has larger default text settings or zooms to 200%, the content remains readable and appropriately laid out. Important text is never locked in an image; it‚Äôs actual text (for example, we wouldn‚Äôt put an important instruction as only part of a logo image, it would also be in text).
  • Color & Contrast: The color scheme is tested for contrast ratios. Primary text is a dark grey on white which is high contrast (meets AA standards). The blue and green accents are used in ways that still contrast against white or grey backgrounds (for instance, white text on the blue buttons is easily readable; blue text on white is used sparingly and typically in larger headings). We avoid color as the sole means of conveying information. Wherever color indicates status (green vs red highlights, etc.), there is also an icon or text label reinforcing it. For example, a ‚ÄúPending‚Äù status might be orange but also explicitly say ‚ÄúPending‚Äù next to it.
  • Keyboard Navigation: The web app is operable with keyboard alone. Interactive elements (buttons, links, form fields) are focusable in a logical order. We‚Äôve defined focus styles (a visible outline or highlight on focused buttons/fields) to help users see where they are. For example, a recruiter tabbing through the dashboard can jump between the New Check button, filter dropdown, each list item, etc., and activate them with Enter/Space. In the reference chat form, pressing ‚ÄúTab‚Äù focuses the input field (or skip link) and pressing ‚ÄúEnter‚Äù submits an answer (as an alternative to clicking send).
  • Screen Reader Support: We include appropriate ARIA labels and roles to make the experience comprehensible via screen readers:
    • The chat interface announces new questions to screen reader users (using ARIA live regions). For example, when a new question bubble appears, it‚Äôs marked as a live announcement so it‚Äôs read out automatically (‚ÄúQuestion: What are the candidate‚Äôs strengths?‚Äù). This way, a visually impaired reference giver knows a question has been asked without having to find it manually.
    • All form inputs have associated labels. In the chat form, although visually the question itself might act as the label for the text input, we ensure that under the hood the input is labeled with the question text (or referenced via aria-labelledby) so the purpose of the textbox is clear (‚ÄúAnswer to: How would you describe their performance?‚Äù).
    • Images (like logos or icons) have descriptive alt text or aria-labels. For example, the company logo in the email and form has alt text ‚ÄúCompany Logo‚Äù. Decorative icons that don‚Äôt impart new information are either given empty alt (so they‚Äôre skipped) or appropriate role=presentation.
    • Complex elements like the report table have additional context for screen readers (e.g., table headers for columns with reference names, so the screen reader can say ‚ÄúQuestion, Would you rehire? ‚Äì Answer from John: Yes, definitely‚Äù).
  • Time & Feedback: There are no unexpected time limits. Reference providers are not logged out if they take too long, and they can always resume via the same link if needed. Any dynamic updates (like a new question appearing) are done in a way that‚Äôs not disorienting. We avoid content that flashes or moves in a distracting way ‚Äì e.g., a typing indicator in the chat might be shown, but it‚Äôs subtle and not a hazard (no rapid blinking). Also, all interactive feedback is provided in multiple ways: for example, when clicking the Submit button, not only does the button change state (‚ÄúSubmitting...‚Äù disabled state), but also an aria-live message might announce ‚ÄúSubmitting, please wait‚Äù, so users know something is happening.
  • Tested Scenarios: We test the UI with different accessibility settings:
    • High Contrast Modes (on Windows, etc.) to ensure our text is still visible (we rely on native OS rendering of buttons and form controls where possible, which adapt to these modes).
    • Zoom/Text Enlargement on mobile (iOS/Android accessibility text size settings) to ensure nothing is cut off or overlapping.
    • Screen reader navigation, as mentioned (with NVDA, JAWS, VoiceOver) to make sure the reading order and labels make sense in context.
    Additionally, the design accommodates translation/localization in the future (for post-MVP), e.g., UI elements are coded to support longer text labels without breaking, even though v1 is English-only.

By embedding accessibility from the start, we not only accommodate users with disabilities but also improve overall UX (often, the app becomes more robust and user-friendly for everyone). The responsive design approach ensures that whether someone is on a large desktop screen, a tablet, or a small smartphone, they can interact with RefCheck with ease. This is particularly important because references might open the questionnaire on their phone during a break, or a hiring manager might quickly check a report on a tablet. We aim to provide a first-class experience on all such platforms.

Interaction & Feedback

Real-time responses and system feedback to user actions

To make RefCheck feel responsive and intuitive, the UI provides appropriate feedback for user actions and incorporates subtle animations where useful. We want users to always understand what‚Äôs happening (or if something is happening in the background) and feel confident that the system is responsive to their input. Here are some key interaction design elements:

  • Button States & Transitions: All clickable elements have visual state changes. For instance, buttons use hover effects (a slightly darker shade of blue on hover for the primary button, and a subtle drop-shadow to indicate elevation) and an active state (perhaps a slight shrink or depress animation when clicked). This gives immediate feedback that the click was registered. After clicking a form submission button, it may turn into a loading state (with a spinner icon and label ‚ÄúSending...‚Äù or ‚ÄúSubmitting...‚Äù) to indicate the system is processing. Once done, it might transform into a success state (for example, turning green with a checkmark) briefly before either the next UI change or disabling entirely to prevent double submission.
  • Loading Indicators: Whenever a background action takes more than a second, a loading indicator is shown. For example:
    • When the dashboard first loads data, a spinner or skeleton screens (grey placeholders) appear in the list until the data populates. This prevents the user from staring at a blank screen or thinking nothing is happening.
    • When the reference chat moves to the next question, there‚Äôs a deliberate but tiny delay to simulate a human-like response; we show three blinking dots ‚Äútyping indicator‚Äù in the assistant‚Äôs bubble for, say, 0.5‚Äì1 second before the next question bubble appears. This not only makes the experience feel conversational but also avoids overwhelming the reference if the next question appears instantaneously.
    • If generating the AI summary on the report takes a moment, the UI might show a placeholder like ‚ÄúAnalyzing responses...‚Äù with a small loading spinner in the summary area, then replace it with the generated text. This way, the recruiter sees that analysis is in progress and doesn‚Äôt think the section is missing.
  • Animations & Transitions: We use animations to create a smooth experience, but always in a subtle, non-distracting manner:
    • Modal dialogs (like the ‚ÄúNew Reference Check‚Äù form) slide down or fade in when opened, focusing the user‚Äôs attention, and conversely fade out when closed. This is quick (around 200ms) to avoid feeling sluggish.
    • Accordions or collapsible sections (if any in the report or templates) animate open/closed so content doesn‚Äôt jarringly appear or disappear.
    • Chat bubbles in the reference form slide up into view as they appear, similar to incoming messages on a phone. This not only looks natural but also provides a visual cue associated with new content. The scrolling container automatically scrolls to the newest message with a smooth animation, so the reference isn‚Äôt required to scroll manually; this is important on mobile to keep the latest question in view when the keyboard opens.
  • Error Handling & Messages: If something goes wrong or input validation fails, the UI clearly informs the user:
    • Form Validation: If the recruiter leaves a required field blank when creating a reference check, that field‚Äôs border highlights in red and an inline message appears (e.g., ‚ÄúEmail is required‚Äù). Error messages are written in plain language and, if possible, provide a remedy (‚ÄúPlease enter a valid email address‚Äù). We ensure focus moves to the first error so screen reader and keyboard users are alerted to it.
    • System Errors: If a network error occurs (unable to load data or send an invite), a notification banner or toast appears with an apology and a retry suggestion. For example, ‚ÄúUnable to send invitations. Check your connection and try again. [Retry]‚Äù. These are often styled as an alert (e.g., with an orange or red left border, and maybe an error icon) and appear at the top or bottom of the relevant area. Persistence: errors that require user action remain visible until dismissed or resolved.
    No error should be silent; the user should never be left guessing why something didn‚Äôt happen. Conversely, once an error condition is resolved (e.g., user corrected the field), the error highlight is removed instantly to confirm that it‚Äôs fixed.
  • Notifications & Confirmations: The system provides confirmations for key actions:
    • After a recruiter sends out reference requests, a brief success toast might appear: ‚ÄúReference requests sent to 3 contacts.‚Äù Similarly, when all references for a candidate are received, an informational notification could appear on the dashboard or via email to the recruiter: ‚ÄúAll references for [Candidate] are in! View the report.‚Äù In the UI, such notifications can be small and auto-dismiss after a few seconds, so as not to clutter, but are noticeable enough (sliding in from a corner or appearing under the header).
    • Some potentially destructive actions (if allowed in UI), like deleting a reference check or removing a reference contact, would prompt the user for confirmation in a modal: ‚ÄúAre you sure you want to remove this reference from the request?‚Äù to prevent mistakes. These confirmation dialogs are clear about the consequences and provide a cancel option.
  • Real-Time Updates: Where possible, the UI updates in real-time without requiring a full page refresh:
    • If the recruiter is viewing a candidate‚Äôs report while a reference submits their form, the system could dynamically load the new response into the report (perhaps with a subtle highlight on new content). At minimum, if the recruiter reopens the report, it will fetch the latest data. The design accounts for partial data as well ‚Äì e.g., if only 2 out of 3 references responded, the report clearly shows which reference is still pending (maybe a placeholder ‚ÄúAwaiting response from [Name]‚Äù). The interface might also subtly encourage action, like showing a ‚ÄúSend Reminder‚Äù link next to that pending reference.
    • The dashboard list auto-refreshes at intervals or via push notifications. If a reference completed a survey, the candidate‚Äôs entry might update from ‚Äú1/2 complete‚Äù to ‚Äú2/2 complete‚Äù and change color (perhaps from in-progress orange to completed green) without the recruiter needing to press refresh. A small blink or fade animation on the updated entry can draw attention to the change.

All these interaction details contribute to making RefCheck feel snappy, modern, and user-friendly. We avoid gratuitous animations that waste time, focusing instead on those that provide feedback or guide the user‚Äôs attention. By managing states (hover, active, loading, success, error) carefully, we ensure the user always knows what‚Äôs happening. For example, a reference provider will see the chat typing indicator and know the next question is coming; a recruiter will see a disabled grayed-out ‚ÄúSend‚Äù button after clicking, so they won‚Äôt accidentally resend a request twice.

In essence, the UI tries to anticipate user needs: confirming actions when they‚Äôre done, validating input as it‚Äôs entered, hinting at next steps, and gracefully handling any hiccups. This anticipatory design reduces frustration and builds trust ‚Äì users feel the app is reliable and ‚Äúhas things under control,‚Äù allowing them to focus on the content of references rather than the mechanics of the software.

In summary, everything the dev team needs to accurately build the UI ‚Äì from exact hex colors to example content ‚Äì is provided.
