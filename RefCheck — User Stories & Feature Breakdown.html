Overview

Translating PRD features into actionable user stories

This document breaks down RefCheck’s product requirements into specific user stories – concise descriptions of desired functionality from an end-user’s perspective. Each user story is accompanied by acceptance criteria that define when the story is considered implemented successfully. These stories serve as a bridge between the high-level Product Requirements Document (PRD) and the actual engineering tasks, ensuring the development team clearly understands the scope and expectations of each feature.

The user stories are grouped by major feature areas of RefCheck, aligning with the MVP features and user personas identified in the PRD. Our primary personas, Rachel (Recruiter) and Mark (Small Business Owner), inform many of these stories, as do the perspective of a reference provider using the system. Each story follows the format: As a [type of user], I want [some goal] so that [benefit]. This keeps the focus on delivering user value.

Acceptance Criteria – for each story, we list specific conditions that must be met for the feature to be accepted. Think of these as checklist items or scenarios that QA will test. They often cover normal cases as well as edge cases (e.g., “what if the reference’s email is invalid?” or “what if the AI cannot generate a summary?”). Meeting all acceptance criteria means the story is done and the feature should satisfy the intended user need without issues.

By implementing all the following stories, we will effectively cover the functionality described in the PRD (v1.0), delivering a cohesive product. The development team can take these stories into sprint planning, estimate them, and build iteratively while ensuring nothing falls through the cracks. Let’s dive into the breakdown:

SI

Setup & Initiation

Creating and configuring reference checks

Feature scope: These stories cover how a recruiter or hiring manager starts a new reference check process, configures the questionnaire, and initiates sending requests. It includes template selection and any preliminary steps like giving candidates a heads-up or using a demo mode.

  • As a recruiter, I can create a new reference check for a candidate in minutes (entering candidate info and their references’ contact details), so that I can quickly kick off the vetting process without delays.
    • Acceptance Criteria:
      • The system provides a clear form to input candidate name, role, and at least 2 reference contacts (name, email, relationship).
      • Form can dynamically add more reference fields (up to a sensible limit, e.g., 5) in case I have more references to check.
      • After submission, a confirmation is shown (e.g., “Reference requests sent for Alice Johnson”) and the new reference check appears on my dashboard with status “In Progress”.
  • As a recruiter, I can choose or customize the questionnaire template for a reference check, so that the questions asked are relevant to the role.
    • Acceptance Criteria:
      • The system offers a default “Standard 5-question Template” by default, but I have an option to select other templates (if available) from a dropdown or list when initiating the check.
      • I can preview the questions in a template before confirming. (For example, seeing the list of 5 questions that will be sent.)
      • If I need a custom question, I can add or edit questions in a simple interface (P1 scope: at least one custom question can be inserted). The system validates that custom questions are not blank and are answerable with the available input types.
      • If no template changes are made, the default template is used. Template selection is remembered per user for convenience (nice-to-have).
  • As a recruiter, I want the system to ensure reference contact info is valid before sending requests, so that I don’t mistakenly send a request to a wrong or malformed email.
    • Acceptance Criteria:
      • If I enter an email in an invalid format (e.g., missing “@” or domain), the form should flag it and prevent me from submitting until corrected.
      • If I accidentally duplicate the same email for two reference entries, the system warns me (but still lets me proceed if I confirm, in case two different people share an email, though unlikely).
      • If the company domain has a blocklist (say, personal emails not allowed), the system warns accordingly. (This criterion might be skipped for v1 unless such rules are defined.)
  • As a new user, I can try out RefCheck in a demo mode with sample data, so that I understand what the final report looks like without having to wait for real references.
    • Acceptance Criteria:
      • A “Launch Demo” or similar button is available on first login or in the dashboard. Clicking it creates a fake candidate (e.g., “John Doe – Demo”) with two sample references.
      • The system auto-fills sample answers and immediately marks the reference check complete, generating a sample report I can view.
      • The demo is clearly labeled (so I won’t confuse it with real data), and it does not send any emails. It can be deleted easily after viewing.
  • As a recruiter, I can configure when and how reference requests are sent, so that I maintain control over timing (e.g., send immediately or schedule for later).
    • Acceptance Criteria:
      • By default, upon creating a reference check, emails are sent immediately to each reference. There is a visible confirmation of this (perhaps listing “Invited on DATE, will resend in 2 days if no response”).
      • (P1) I have the option to schedule the send (for example, I could set “Send invitations on Monday 9am” if I’m setting it up on a weekend). If scheduled, the system queues them and clearly shows the scheduled time.
      • The system automatically handles sending a reminder email at the set interval (e.g., 2 days later) if a reference hasn’t responded. This behavior is either default or configurable globally (for v1, it can be default behavior). I can see this setting in the UI (even if it’s just informational in v1).
RF

Reference Feedback Collection

Gathering input via a conversational form

Feature scope: These stories capture the experience of the reference who receives the request. The goal is to make it easy, quick, and clear for them to provide feedback. It covers the email invite, the secure web form, and the conversational interface, including any guidance or ability to skip questions.

  • As a reference provider, I receive a clear email request for feedback, so that I understand who it’s for and trust that it’s legitimate.
    • Acceptance Criteria:
      • The invitation email includes the candidate’s name and the company’s name in the subject and greeting (e.g., “Reference Request for Alice Johnson at ABC Corp”).
      • The email content briefly explains what is needed (e.g., “Alice is applying for X role and listed you as a reference. Please click the link to answer a few questions.”) and assures confidentiality of their answers.
      • The email is branded (company logo or name visible, and “Powered by RefCheck” subtly). The sender address is a trusted domain and not a random string (to avoid looking like spam).
      • The email contains a prominent link or button which leads to the secure questionnaire. If possible, it also mentions approximately how long it will take (~5 minutes).
  • As a reference provider, I can access the feedback form without any login or hassle, so that I’m not discouraged from completing it.
    • Acceptance Criteria:
      • Clicking the link in the email opens the RefCheck reference form immediately in a browser. There is no account creation or password needed; the secure token in the URL authenticates me.
      • The page greets me by my name (“Hi Bob, thanks for helping with a reference for Alice.”) and reiterates the purpose. It also shows the company and candidate name clearly, so I feel confident about what I’m doing.
      • If the link has already been used (survey submitted), it should inform me that I’ve already submitted feedback (to prevent duplicate entries or confusion).
      • The form is mobile-friendly – if I open it on my phone, fields and buttons are easy to use and don’t require horizontal scrolling.
  • As a reference provider, I am guided through a conversational Q&A form, so that providing feedback feels like a quick, friendly chat rather than a tedious survey.
    • Acceptance Criteria:
      • The interface presents one question at a time, in a chat bubble or similar format, with an obvious way to answer (text box for open-ended, multiple-choice buttons for yes/no or rating scale for numerical questions).
      • After I submit an answer, I immediately see it recorded (e.g., my answer appears as a chat bubble from “me”) and the next question appears. There’s minimal lag; any loading (like an AI follow-up) is indicated by a “typing…” placeholder for a second or two.
      • If I give a very short answer to an open-ended question (like just a few words), the system prompts me politely, “Could you share a bit more detail?” This prompt is optional to follow – I can ignore it and proceed, but it encourages richer feedback.
      • A progress indicator or step count (e.g., “Question 3 of 5”) is visible, so I know how many more questions remain.
  • As a reference provider, I can skip a question or save and resume if needed (though the process is short), so that I’m not forced to answer something I’m uncomfortable with or if I’m interrupted.
    • Acceptance Criteria:
      • For each question, there is a “Skip” option visible. If I click skip, the form records a null answer (and possibly asks me to confirm skipping if it’s not the last question, to avoid accidental skips).
      • If a question is skipped, the app might ask for a reason (optional) or simply note “(Reference chose to skip this question)” for the recruiter’s report.
      • If I close the form midway, and later click the same email link, the form remembers my progress and I continue from where I left off (using the same secure token). This requires saving partial answers on our end.
      • When resuming, I’m not asked the same questions again (unless my prior answers didn’t save due to, say, a network issue – in which case a graceful message might ask me to re-answer that one).
  • As a reference provider, I review and submit my feedback at the end, so that I have a chance to correct anything before it’s final.
    • Acceptance Criteria:
      • After the last question, I see a summary of all my answers on one page or in the chat scroll. I have the ability to edit any answer (for instance, an “Edit” button next to each response that jumps me back to that question or opens a text box to change it).
      • Once satisfied, I click a final “Submit” button. If I try to leave before clicking Submit (with all questions answered or skipped), the system warns “Your feedback isn’t submitted yet. Are you sure you want to exit?” to prevent accidental abandonment.
      • Upon submission, I see a thank-you confirmation message. It might say “Thank you for your time! Your feedback has been submitted.” and possibly reassure “The hiring team will use this information confidentially.”
      • (Optional) A very short satisfaction survey appears (1-5 star experience rating, and/or a comment box). This is clearly optional. If I choose to fill it, great, if not, I can simply close the page. Submitting this does not affect the reference feedback submission which is already done.
AR

Analysis & Review

AI processing and recruiter consumption of results

Feature scope: These stories cover what happens once references have submitted their feedback: the AI analysis of the textual answers, generation of summary reports, and how the recruiter accesses and uses that information. It ensures that the insights are delivered accurately and highlight what’s important, assisting the recruiter in making decisions.

  • As a recruiter, I am notified when a reference check is complete with all responses, so that I can review the feedback promptly.
    • Acceptance Criteria:
      • Once all invited references have submitted their feedback (or the required minimum number), I receive an email notification saying “Reference Check Complete for [Candidate Name]”. The email contains a link to view the report and perhaps a quick summary if safe to include.
      • If I’m logged into the dashboard, I also see a real-time update: the status for that candidate changes from “In Progress” to “Completed” (perhaps highlighted in green), and a icon or badge indicates new results available.
      • If some references responded but not all, I can still click into the interim report which will show who is pending. I’m notiﬁed in the UI that it’s waiting on others. A “Resend invite” or “Add another reference” action is available if needed (outside scope of pure notification, but part of review flow).
  • As a recruiter, I can view a consolidated report of all reference feedback with key insights highlighted, so that I get an overview of the candidate’s strengths and weaknesses at a glance.
    • Acceptance Criteria:
      • The report begins with an AI-generated summary that accurately reflects the tone and content of the references. (E.g., “Both references praised X. One mentioned Y as an area to watch.”) The language is professional and clear, and ideally concise (no more than a few paragraphs or bullet points).
      • The report lists each question asked and groups the answers from each reference under that question. For example, Q1 with Reference A’s answer and Reference B’s answer below it. This makes it easy to compare responses side by side.
      • Any negative or concerning comments in answers are automatically highlighted (yellow for moderate concern, red for serious). For instance, if a reference wrote “would hesitate to rehire,” that phrase is marked in red. If they wrote “no major issues, however sometimes tardy,” the tardiness remark might be highlighted in yellow. The highlights include a footnote or tooltip explaining them (like “flagged by AI for negative sentiment”).
      • The overall sentiment or a simple rating per reference is shown (perhaps an icon or score). For example, a smiling icon next to Reference A (who was very positive) and a neutral icon next to Reference B (who was more mixed), helping me quickly gauge each reference’s stance without reading every word first.
      • I can download the report as a PDF, and the PDF retains the formatting and highlights, including company branding if applicable.
  • As a recruiter, I want the AI analysis to be accurate and adjustable, so that I trust the insights and can correct them if needed.
    • Acceptance Criteria:
      • The AI summary should not invent facts not present in the feedback. (Test: if references didn’t mention a trait, the summary shouldn’t introduce it.) It should use wording that a human recruiter would use. We’ll have an internal validation step (perhaps the first few uses ask the recruiter “Is this summary accurate?” with a thumbs up/down input to gather feedback).
      • If I mark a summary as inaccurate or notice a misflagged comment, I have a way to give that feedback (e.g., clicking a thumbs-down triggers a text box “What was wrong?”). This is logged for future improvements, but in the immediate term the system might remove a flag or adjust the summary if possible. (v1 might not dynamically change it, but at least logs the input.)
      • The system errs on the side of caution: it does not flag things that are only subtly negative as major issues. For example, if a reference answer is mostly positive with one minor critique, the summary reflects that nuance (“mostly positive with a minor critique about...”) and perhaps no highlight if it’s truly minor. Only clear negative phrases or themes are highlighted/flagged.
      • The AI module respects data privacy: it does not leak content from other candidates or irrelevant info. (This is more an internal criteria: ensure prompts and API use only the relevant text.)
  • As a recruiter, I can identify and follow up on any red flags easily, so that I can address concerns with the candidate or references before hiring.
    • Acceptance Criteria:
      • If any reference answer triggered a serious concern (like mention of misconduct, poor performance, or a clear “would not rehire”), the system explicitly calls that out in the report. Possibly a special section “⚠ Red Flags” that lists the issues or at least arrows pointing to the highlighted text in answers.
      • The report or UI may offer suggested follow-up questions for these flags. For example, if punctuality was an issue, a note might say “Consider asking the candidate about their time management and how they’ve improved it in subsequent roles.”
      • I can add a comment or note on the report for internal record (e.g., “Discussed with candidate on 2025-08-04, they provided context XYZ”). This note gets saved with the report and possibly included in the PDF export if I choose.
      • (Optional, P1/2) If multiple references gave notably different feedback (one glowing, one very lukewarm), the system highlights that discrepancy – e.g., a line in summary like “References provided mixed feedback: one was strongly positive, another had reservations about...”. This ensures I don’t miss inconsistency between responses.
  • As a hiring manager (viewer), I can read the reference report shared with me without editing it, so that I’m informed on the candidate’s background while maintaining data integrity.
    • Acceptance Criteria:
      • When a recruiter shares a completed report (via adding me as a Viewer or sending a secure link), I can open it and see exactly what the recruiter sees (summary, answers, flags) but I have no option to edit or add notes. The UI clearly is read-only for me.
      • I do not see the reference providers’ contact info (if the recruiter opted to anonymize it for sharing). For instance, it might label them “Reference 1 – Former Manager” instead of “Bob Manager, bob@example.com” to preserve confidentiality when sharing beyond HR.
      • If I have questions or comments, I rely on our internal process (e.g., I might call the recruiter or add a note if the system allows input from viewers that’s separate). But by default, the viewer role is just consumption. The system might show a watermark or note “Internal Use Only” to remind that this info is sensitive.
      • The act of sharing is tracked (audit log: e.g., “Report viewed by John HiringManager on Aug 5”). This is not visible on the frontend to me as viewer, but it’s an acceptance criterion on the backend for compliance.

Tracking & Management

Dashboard, status, and ongoing management of checks

Feature scope: These stories are about the recruiter’s ability to track multiple reference checks, see statuses at a glance, and manage ongoing checks (like sending reminders or handling incomplete responses). It ensures the tool provides visibility and control throughout the process, not just at the end.

  • As a recruiter, I can see all my ongoing and completed reference checks in a dashboard, so that I have a single view of where each candidate stands in the process.
    • Acceptance Criteria:
      • The dashboard lists each reference check (likely by candidate name and role) with a status indicator (e.g., “2/3 references completed” or “Completed” or “Waiting on 1 response”). It’s sorted by most recent or those needing attention (e.g., incomplete ones at top).
      • Each entry shows key info: candidate name, position, date initiated, and perhaps a visual progress (like a mini progress bar or fraction as text). If an invite bounced or was declined, that could reflect (e.g., an exclamation icon) to draw my attention.
      • I can filter or search this list (for example, search by candidate name or filter to “Completed last 30 days” vs “In Progress”). In v1, even a simple text search is fine for finding a candidate quickly.
      • The design is responsive: on a wide screen, it may show multiple columns (Name, Role, Status, Date); on mobile, it might collapse to a card per candidate for scrolling.
  • As a recruiter, I can click into a specific reference check to view more details or take actions, so that I can manage each case individually.
    • Acceptance Criteria:
      • Clicking a candidate entry opens a detailed view (or drawer/modal) showing that candidate’s reference check progress: list of references (with name, email, relation, status each). For each reference, I see whether they’ve opened the invite (if we track email opens) and whether they’ve submitted feedback.
      • If a reference is pending and, say, 3 days have passed, I see an option to “Resend Reminder” next to that reference. Clicking it triggers an immediate reminder email and the UI confirms (“Reminder sent to Carol on Aug 5”).
      • I can also edit a reference’s email if I realize there was a typo (e.g., change .con to .com). Editing it triggers sending the invite to the new address (and possibly voiding the old link). This action is logged and only allowed if that reference hasn’t responded yet.
      • If a reference declines or can’t be reached, I have the option to add a new reference to replace them. (P1 feature: “Add another reference” which would send a fresh invite to a new contact while marking the old one as canceled.)
  • As a recruiter, I receive alerts if a reference check is stalled or an email bounces, so that I can proactively intervene.
    • Acceptance Criteria:
      • If 3 days pass and a reference hasn’t clicked their link, the system generates an alert. This could be an email (“Reminder: 1 of 3 references for Alice hasn’t responded yet. Consider reaching out personally.”) and/or a highlight on the dashboard (the candidate entry might turn orange or have a warning icon).
      • If an invitation email bounces (invalid address) or is undeliverable, I am notified within the day. The candidate’s detail view will mark that reference as “Email failed – invalid address”. I can then correct the email and resend.
      • If after X days (e.g., 7 days) not all references are in, I receive a summary email listing those still pending across all my checks. (This might be a digest, perhaps P1, to catch anything that slipped through.)
      • All such alerts can be toggled in settings if needed (but default is on to help busy recruiters).
  • As a recruiter, I can mark a reference check as complete or cancel it manually if needed, so that I can stop the process when it’s no longer needed.
    • Acceptance Criteria:
      • If I decide to hire or disqualify a candidate before all references come in, I can mark that reference check as closed. This might appear as a “Cancel Reference Check” or “Mark as Complete” button in the detail view.
      • Upon canceling, any pending reference links are deactivated (if they click, they might see “This reference request is no longer active.”). The dashboard marks it accordingly (maybe a status “Canceled” or simply removes it from active list).
      • Canceled checks are not counted in metrics like completion rate (or counted separately) to avoid skewing data.
      • I can still view partial reports for canceled checks (if some references had responded) in an archive section, but it’s clearly labeled as incomplete/canceled. This allows retrieval if needed, but mainly it’s for record-keeping.

Collaboration & Permissions

Multi-user support and access control

Feature scope: These stories deal with multiple users within the hiring organization using RefCheck. It outlines what an Admin can do versus a regular recruiter, and how a recruiter can share reference check results with others in a secure way. While some of these are P1 enhancements, it’s important to design with them in mind for future growth (especially for teams).

  • As an admin user, I can invite colleagues to use RefCheck under our company account, so that our hiring team can collaborate and share workload.
    • Acceptance Criteria:
      • There’s a “Users” or “Team” management section where I (admin) can add a new user by entering their email and assigning a role (Recruiter or Viewer).
      • Inviting sends that person an email with a link to set up their password and join the account. Until they accept, their status might show as “Pending” in the user list.
      • Admin can remove a user or change their role. Removing immediately revokes access (that user’s login can no longer access our company data). Changing role updates what they can see and do (e.g., demoting an admin to viewer or vice versa).
      • For v1, we assume one admin (the account creator) and others as recruiters or viewers. All recruiters can see all reference checks for that company (no per-recruiter silo yet), unless decided otherwise. This is communicated to users to avoid confusion.
  • As a recruiter, I can share a reference report with the hiring manager via a secure link or add them as a viewer, so that they can review the feedback themselves.
    • Acceptance Criteria:
      • On a completed report page, I have an action to “Share Report”. Options include: generate shareable link or invite user.
        • Shareable Link: Clicking generates a unique URL that I can copy. Anyone with this URL can view the report (no login) but it’s obfuscated (long random token). I can set an expiry (default 7 days) for security. I can also disable the link if needed (which then invalidates it).
        • Invite Viewer: I enter an email, the system sends them an invite to create a viewer account limited to this report or all reports? (Likely simpler: add as full viewer user on account, then use RBAC to limit if needed.) For v1, maybe skip granular per-report permission; assume shareable link for quick sharing, viewer accounts for ongoing access.
      • The shared report (via link or viewer login) does not allow the viewer to access other candidate reports unless they’ve been given those too. If using link, it’s only that report. If using a viewer account (with role-based), maybe viewers can see all completed reports by default or ones an admin assigns. Clarity needed – likely: make them a viewer user = can see all, which might be acceptable for a small team scenario.
      • Audit logging: It’s recorded when a share link is created and by whom, and if it’s used to access the report (timestamp, maybe IP/browser for security). Similarly, adding a viewer user is logged. These logs are available to admins upon request (not necessarily in UI v1, but stored for compliance).
      • (Security) The share link should not expose reference identities if not desired. Perhaps by default it hides reference names and contact, labeling them generically (given the viewer may sometimes be the hiring manager who might know the candidate personally or even the candidate if share accidentally - though we wouldn’t share with candidate in this flow, but just in case of leak, minimize identifiable info).
  • As a basic user (recruiter), I cannot alter account-wide settings or view others’ private info (like billing), so that sensitive settings remain controlled by admin.
    • Acceptance Criteria:
      • In the UI, non-admin recruiters do not see the “Team Management” or “Billing” sections. They also cannot delete data outside their scope.
      • If a non-admin tries to access an admin endpoint (e.g., via direct URL), the server returns 403 Forbidden. The UI may show a polite “You do not have access to this area.”
      • Regular recruiters can create and manage reference checks (that’s their role’s scope). If in future we implement a setting where recruiters only see their own candidates, that would be handled by linking candidates to the creator – for v1, all recruiters share a pool as mentioned, so this criterion is more about not letting them manage high-level settings.
      • Viewer users basically have read-only access. They cannot create or edit reference checks. The UI hides all buttons for sending or editing from them. If a viewer user somehow invokes a change action, it’s rejected by the backend as well. Essentially, viewer sees dashboard and reports but with no actionable controls.
  • As an admin, I can configure global settings like default reminder schedule or data retention preferences ((P1/2 for some settings)), so that the system aligns with our company’s policies.
    • Acceptance Criteria:
      • In a “Settings” page, options might include:
        • Reminder cadence: e.g., send first reminder after 2 days (toggle on/off or set number of days).
        • Auto-close: e.g., mark checks as done after X days regardless of responses (maybe not v1, but a policy some might want).
        • Data retention: e.g., automatically archive/delete reference data after N years (for compliance). For v1, likely just stating data will be kept 2 years by default, but giving an option if needed for strict companies.
      • These settings changes affect future behavior (e.g., if I change reminder interval, the next unsent reminders follow the new schedule; existing waiting ones might adjust accordingly).
      • Only admins can change settings; recruiters can view them but not edit (or we hide entirely from non-admin to avoid confusion).
      • Changes are confirmed (e.g., “Settings saved” message) and logged (audit: “Reminder interval changed from 2 to 3 days by Admin Alice on Aug 6”).
IA

Integration & Automation

Connecting with external systems and automating workflows

Feature scope: These stories deal with how RefCheck interacts with other tools and automates certain tasks. This includes integration with Applicant Tracking Systems (ATS) like Greenhouse or Lever (most likely via Zapier or webhooks in v1), as well as automated notifications or scheduled reports that go out to stakeholders.

  • As a recruiter, I want RefCheck to integrate with our ATS, so that reference checks can start automatically and results can be seen in the candidate’s ATS profile.
    • Acceptance Criteria:
      • If using Zapier or similar: we have a documented template such that when a candidate is moved to a “Reference Check” stage in the ATS, Zapier calls the RefCheck API to create a reference check (populating candidate name, role, and ideally references if those were collected). This should trigger just as if I manually created it, and I get notified in RefCheck.
      • Alternatively or additionally, we support an incoming webhook from ATS: e.g., ATS sends candidate data to /api/v1/integrations/ats, which our system processes. For acceptance, at least one example integration (like with Greenhouse or Lever via Zapier) is tested and works in v1.
      • After completion, RefCheck can send a webhook or use Zapier to add a note to the ATS: e.g., “Reference check completed on [date]. Summary: ...” and perhaps attach the PDF or link. Criteria: The note appears in the ATS within a short time of completion (say, within 5 minutes) and contains enough info for someone glancing at ATS to know references are done and where to find details.
      • These integration actions are configurable (some users might not want auto-start or auto-post to ATS). In v1, it might require setup via Zapier or an admin enabling webhooks – which is acceptable as a manual configuration step documented in our guide.
  • As a hiring team member, I can receive a daily or weekly summary of reference check activity, so that I’m kept in the loop without constantly checking the app.
    • Acceptance Criteria:
      • If I opt in (or by default), the system sends a daily digest email of reference check statuses. For example: “Today’s RefCheck Summary: 2 new checks initiated (Alice J., Bob K.), 3 completed (Carol L., Dan M., Erin N.), 1 pending (Frank O. waiting on 1 reference).”
      • Likewise, a weekly summary could include stats like “This week you initiated 5 checks, average turnaround time 1.2 days.” That’s more P2, but the daily/weekly option can be combined and formatted nicely.
      • The email only goes to users who want it (toggle in settings or unsubscribe link in the email). It aggregates across the company or just the user’s own candidates depending on scenario – given v1 likely has all recruiters see all, one digest for all might be fine for small teams. If team is larger, might need personalized, but we can keep it simple for now.
      • No sensitive reference content is in the summary – it’s high-level info. Detailed content stays in the app links.
  • As a system owner, I ensure that RefCheck’s API and webhooks support future integrations easily, so that we can integrate with internal tools or other services as needed.
    • Acceptance Criteria:
      • We have API endpoints (authenticated) for the core actions: creating a reference check, retrieving a report, etc., as documented in the API guide. They work as expected (tested with sample calls).
      • There is a webhook mechanism that can send out events: at minimum “reference_check.completed”. It can be configured with a URL in settings or per request via callback_url. This webhook sends the necessary data (reference_check_id, status, summary snippet, etc.) for an external system to consume. We test it with a requestbin or similar to ensure it’s firing and properly formatted.
      • The system can handle the case where an external system calls our API to create a reference check for a candidate that’s already in our database (perhaps from a previous check). Ideally, it should create a new check instance cleanly, not merge with old data except reusing template or user info appropriately.
      • The API and webhook transactions are secure (using SSL, requiring auth for API, and providing a signature or secret for webhooks as possible). This might not be directly user-facing but is part of acceptance for integration-readiness.
  • As a recruiter, I want certain actions to be automated, so that I can focus on decisions rather than administration.
    • Acceptance Criteria:
      • (Many automation points already covered: auto-reminders, auto-notifications, etc. This story ensures we consider any other repetitive tasks.)
        • For instance, auto-forwarding the final report to a particular email distribution (maybe HR team alias) once complete – could be a setting the admin turns on. If on, then acceptance is that an email with PDF goes to the specified address(es) upon completion.
        • Another automation: if a candidate’s references complete and everything is positive (no flags), optionally trigger moving them to next stage via integration. This might be later scope, but we note it as a possibility.
      • No automation should happen without either a reasonable default or explicit configuration. We prefer opt-in for sending data outward. E.g., by default we might not email PDFs externally, but we allow it if set up.
      • Automations are logged and transparent – e.g., if a report was auto-emailed to someone, that appears in an activity log for that reference check (“Report auto-sent to hiring@abc.com”). This is critical for compliance and debugging.
